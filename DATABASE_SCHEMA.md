# KitchenOS — Complete Database Schema
> Supabase (PostgreSQL). Feed this to your Claude project knowledge before working on any data layer.

---

## OVERVIEW

Multi-tenant architecture. Every table except `profiles` carries a `kitchen_id` foreign key.
All reads and writes in the application are scoped to the authenticated user's `kitchen_id`.
Primary keys are UUIDs generated by Supabase (`gen_random_uuid()`).

## TABLES

---

### `kitchens`

The root entity. One row per cloud kitchen. Created at signup.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `kitchen_id` | `uuid` | NOT NULL | Primary key |
| `owner_user_id` | `uuid` | NOT NULL | FK → `auth.users.id` |
| `name` | `text` | NOT NULL | Display name, shown on storefront |
| `slug` | `text` | NULL | URL slug for storefront — `[a-z0-9-]` only. Unique. Used in `/[slug]` routes |
| `email` | `text` | NOT NULL | Business email. Set at signup. Not editable via UI. |
| `phone` | `text` | NULL | Required to complete profile (triggers `ProfileBanner` warning if null) |
| `city` | `text` | NULL | Required to complete profile (triggers `ProfileBanner` warning if null) |
| `welcome_banner` | `text` | NULL | Hero greeting on customer storefront |
| `accent_color` | `text` | NULL | Hex string for storefront header colour (Surface D) |
| `settings` | `jsonb` | NOT NULL | See structure below |
| `created_at` | `timestamptz` | NOT NULL | Auto-set |

**`settings` JSONB structure:**
```json
{
  "notify_channel": "email" | "whatsapp" | "both" | "none",
  "theme_color": "#hex",
  "delivery_radius_km": 5
}
```
Settings are always merged on update (spread existing + overwrite changed keys) — never replaced whole.

**Queries in codebase:**
- `SELECT * WHERE slug = ?` → customer storefront lookup
- `SELECT * WHERE kitchen_id = ?` → settings page
- `UPDATE (name, phone, city, slug, welcome_banner, settings) WHERE kitchen_id = ?` → SettingsForm

### `profiles`

One row per Supabase Auth user. Created at signup alongside the kitchen row.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `user_id` | `uuid` | NOT NULL | PK + FK → `auth.users.id` |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `role` | `text` | NOT NULL | `'owner'` or `'employee'`. Only `'owner'` passes the dashboard auth guard |
| `name` | `text` | NOT NULL | Full name. Set at signup from `firstName + lastName` |
| `phone` | `text` | NULL | Profile phone (not the kitchen phone) |

**Auth guard pattern (used in every dashboard page):**
```ts
const { data: { user } } = await supabase.auth.getUser()   // server-verified
const { data: profile } = await supabase
  .from('profiles')
  .select('kitchen_id, role')
  .eq('user_id', user.id)
  .single()
```

**Dashboard layout guard (role check):**
```ts
// layout.tsx joins kitchens:
.select('*, kitchens(*)')
// Only 'owner' and 'employee' pass:
if (!['owner', 'employee'].includes(profile.role)) redirect('/login')
```

**Signup INSERT sequence:**
1. `supabase.auth.signUp({ email, password })` → creates `auth.users` row
2. `INSERT INTO kitchens (owner_user_id, name, email, settings)` → returns `kitchen_id`
3. `INSERT INTO profiles (user_id, kitchen_id, role='owner', name)`

### `menu_items`

One row per item on a kitchen's menu.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `item_id` | `uuid` | NOT NULL | Primary key |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `name` | `text` | NOT NULL | |
| `description` | `text` | NULL | |
| `category` | `text` | NOT NULL | Free-text string, used for grouping on storefront |
| `price` | `numeric` | NOT NULL | In PKR |
| `is_active` | `boolean` | NOT NULL | Inactive items hidden from customer storefront |
| `image_url` | `text` | NULL | Absolute URL to image |

**Queries in codebase:**
- `SELECT * WHERE kitchen_id = ? ORDER BY category` → MenuBoard (dashboard)
- `SELECT * WHERE kitchen_id = ? AND is_active = true ORDER BY category` → customer storefront
- `UPDATE (is_active) WHERE item_id = ?` → toggle active (MenuBoard)
- `UPDATE (name, category, price, image_url, is_active) WHERE item_id = ?` → edit (MenuBoard)
- `INSERT (kitchen_id, name, category, price, image_url, is_active)` → add new item (MenuBoard)
- `DELETE WHERE item_id = ?` → delete item (fails with FK error if item exists in order_items — blocked by constraint)

### `customers`

One row per unique customer (identified by phone + kitchen_id). Created at order placement.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `customer_id` | `uuid` | NOT NULL | Primary key |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `name` | `text` | NOT NULL | Entered by customer at checkout |
| `email` | `text` | NULL | Optional, entered at checkout |
| `phone` | `text` | NOT NULL | Pakistan format, normalized via `normalizePhone()`. Uniqueness enforced per kitchen via `(kitchen_id, phone)` |
| `total_orders` | `integer` | NOT NULL | Counter. Set to 0 on INSERT. Updated externally (via DB trigger or N8N) |
| `total_spend` | `numeric` | NOT NULL | Running total. Set to 0 on INSERT. Updated externally |
| `first_order_at` | `timestamptz` | NULL | Set externally (trigger or N8N) |
| `last_order_at` | `timestamptz` | NULL | Set externally (trigger or N8N). Used for sorting on Customers page |

**Order placement flow (CartSheet.tsx):**
1. `SELECT customer_id WHERE kitchen_id = ? AND phone = ?` → check if exists
2. If not found → `INSERT (kitchen_id, name, email, phone, total_orders=0, total_spend=0)`
3. Use returned `customer_id` for the order INSERT

**Queries in codebase:**
- `SELECT * WHERE kitchen_id = ? ORDER BY last_order_at DESC LIMIT 100` → Customers page
- `SELECT customer_id WHERE kitchen_id = ? AND phone = ?` → pre-check at order time
- `SELECT *, customers(name) JOIN` on orders → dashboard order tables

### `orders`

One row per customer order.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `order_id` | `uuid` | NOT NULL | Primary key. Also serves as capability token for anonymous tracking |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `customer_id` | `uuid` | NOT NULL | FK → `customers.customer_id` |
| `status` | `text` | NOT NULL | Enum — see status values below |
| `total_amount` | `numeric` | NOT NULL | Sum of `unit_price × quantity` for all items |
| `delivery_address` | `text` | NOT NULL | Free text entered at checkout |
| `notes` | `text` | NULL | Kitchen notes from customer |
| `created_at` | `timestamptz` | NOT NULL | Auto-set |
| `updated_at` | `timestamptz` | NOT NULL | Updated on every status change |

**Status enum (enforced in `lib/types.ts`):**
```
pending → confirmed → preparing → ready → dispatched → out_for_delivery → delivered
                                                                         ↘ cancelled (any point)
```

**Status transition rules (from the codebase):**
- Owner: can set any status
- Employee: can advance `pending → confirmed → preparing → out_for_delivery`; cannot mark `delivered` or `cancelled`

**Queries in codebase:**
- `SELECT *, customers(name,phone,email), order_items(*, menu_items(name)) WHERE kitchen_id = ? AND created_at >= pktStartOfDay` → dashboard + KDS
- `SELECT *, order_items(*, menu_items(name)) WHERE order_id = ?` → order tracking page
- `SELECT *, order_items(*, menu_items(name)) WHERE customer_id = ? ORDER BY created_at DESC LIMIT 50` → customer detail panel
- `SELECT created_at, total_amount, status, order_items(...) WHERE kitchen_id = ? AND status = 'delivered' AND created_at >= 30daysAgo` → Reports
- `INSERT (kitchen_id, customer_id, status='pending', total_amount, delivery_address, notes)` → CartSheet on order placement
- `UPDATE (status, updated_at) WHERE order_id IN (...)` → OrderBoard bulk status update
- `UPDATE (status, updated_at) WHERE order_id = ?` → KDS single card status advance

### `order_items`

Line items for each order. Many-to-one with `orders`.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `order_item_id` | `uuid` | NOT NULL | Primary key |
| `order_id` | `uuid` | NOT NULL | FK → `orders.order_id` |
| `item_id` | `uuid` | NOT NULL | FK → `menu_items.item_id` |
| `quantity` | `integer` | NOT NULL | |
| `unit_price` | `numeric` | NOT NULL | Snapshot of price at time of order (not live from menu_items) |

**Note on `item_id` naming:** The TypeScript type uses `item_id`, but the codebase also references `menu_item_id` in some join results. The actual PK on `menu_items` is `item_id`.

**INSERT at order placement (CartSheet.tsx):**
```ts
const orderItems = cartItems.map(item => ({
  order_id: newOrder.order_id,
  item_id: item.item_id,
  quantity: item.quantity,
  unit_price: item.price,   // snapshot at checkout time
}))
await supabase.from('order_items').insert(orderItems)
```

**Joined in queries as:**
```ts
order_items(*, menu_items(name))
order_items(quantity, unit_price, menu_items(name))
```

### `notifications_log`

Stores automation events queued for owner approval (low stock reorders, win-back campaigns, upsell offers).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `notification_id` | `uuid` | NOT NULL | Primary key |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `type` | `text` | NOT NULL | `'low_stock'` \| `'win_back'` \| `'upsell'` \| `'supplier_message'` \| `'menu_disable'` |
| `payload` | `jsonb` | NOT NULL | Type-specific data. For `low_stock`: includes ingredient name, supplier name/phone |
| `status` | `text` | NOT NULL | `'pending'` \| `'approved'` \| `'rejected'` \| `'auto_executed'` |
| `approval_token` | `uuid` | NOT NULL | Unique token passed to N8N webhook on approval |
| `created_at` | `timestamptz` | NOT NULL | Auto-set |
| `resolved_at` | `timestamptz` | NULL | Set when approved or rejected |

**Queries in codebase:**
- `SELECT COUNT(*) WHERE kitchen_id = ? AND status = 'pending'` → Sidebar badge + Overview stat card
- `SELECT COUNT(*) WHERE kitchen_id = ? AND type = 'low_stock' AND status = 'pending'` → Overview "Low Stock" stat card
- `SELECT * WHERE kitchen_id = ? AND status = 'pending' ORDER BY created_at DESC` → Approvals page
- `UPDATE (status, resolved_at) WHERE notification_id = ?` → ApprovalsBoard on approve/reject

**Approval flow:**
1. ApprovalsBoard calls `POST /api/approve` with `{ token: notification.approval_token }`
2. API route verifies owner auth, calls N8N webhook
3. On success, ApprovalsBoard calls `UPDATE notifications_log SET status='approved', resolved_at=now()`
4. On reject (no N8N call), directly `UPDATE SET status='rejected'`

### `feedback`

Customer ratings submitted after delivery. Submitted via N8N webhook proxy.

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `feedback_id` | `uuid` | NOT NULL | Primary key |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `order_id` | `uuid` | NOT NULL | FK → `orders.order_id` |
| `customer_id` | `uuid` | NOT NULL | FK → `customers.customer_id` |
| `rating` | `integer` | NOT NULL | 1–5 |
| `comment` | `text` | NULL | Max 500 chars (validated in API route) |
| `created_at` | `timestamptz` | NOT NULL | Auto-set |

**Important naming note:** The table name in the database is `feedback` (singular). The Reports page has a bug referencing it as `feedbacks` (plural) — this query always returns empty. The correct table name is confirmed by the customer storefront page which queries `feedback` successfully.

**Queries in codebase:**
- `SELECT *, customers(name) WHERE kitchen_id = ? ORDER BY created_at DESC LIMIT 6` → customer storefront (recent reviews section)
- `SELECT rating WHERE kitchen_id = ?` → Reports page (bug: queries `feedbacks` — should be `feedback`)

**Submission flow:**
1. Customer submits at `/[slug]/feedback`
2. Page calls `POST /api/feedback` (Next.js proxy route)
3. API validates with Zod, forwards to `N8N_WEBHOOK_URL/feedback`
4. N8N is responsible for writing the row into Supabase (NOT the Next.js app directly)

### `stock_logs`

Audit trail of every stock change (decrements from wastage; increments from deliveries).

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `log_id` | `uuid` | NOT NULL | Primary key |
| `kitchen_id` | `uuid` | NOT NULL | FK → `kitchens.kitchen_id` |
| `ingredient_id` | `uuid` | NOT NULL | FK → `ingredients.ingredient_id` |
| `change_amount` | `numeric` | NOT NULL | Negative for wastage/consumption, positive for restocking |
| `reason` | `text` | NULL | Free text. Defaults to `'wastage'` if blank |
| `logged_by` | `uuid` | NOT NULL | `auth.users.id` of the staff member who logged it |
| `created_at` | `timestamptz` | NOT NULL | Auto-set |

**INSERT at wastage logging (WastageForm.tsx):**
```ts
await supabase.from('stock_logs').insert({
  kitchen_id,
  ingredient_id,
  change_amount: -parsedAmount,   // always negative for wastage
  reason: reason || 'wastage',
  logged_by: profileId            // user.id passed from server
})
// Then separately:
await supabase.from('ingredients').update({ current_stock: newStock }).eq('ingredient_id', ...)
```
Note: These are two separate writes, not a transaction. If the second fails, stock_logs will have a phantom entry.

## FUNCTIONS (RPC)

### `find_order_by_prefix`

Used by the customer-facing Track Order panel to support short 8-character IDs alongside full UUIDs.

```sql
find_order_by_prefix(p_prefix text, p_kitchen_id uuid)
RETURNS TABLE(order_id uuid)
LANGUAGE sql
SECURITY DEFINER
AS $$
  SELECT order_id FROM orders
  WHERE kitchen_id = p_kitchen_id
    AND order_id::text ILIKE (p_prefix || '%')
  LIMIT 1;
$$;
```

**Called from (MenuPageClient.tsx):**
```ts
const { data } = await supabase
  .rpc('find_order_by_prefix', {
    p_prefix: raw.toLowerCase(),
    p_kitchen_id: kitchen.kitchen_id
  })
```

**Important:** The function was previously broken (wrong return type) and has been fixed. The `DROP FUNCTION IF EXISTS` must precede any recreation because PostgreSQL does not allow changing a function's return type in-place.

## ROW-LEVEL SECURITY (RLS)

### Current state (as of last audit)

RLS is enabled on all tables. However, some policies have security weaknesses that are pending manual SQL fixes in the Supabase dashboard.

#### Tables with open policies (PENDING FIX):

**`customers`, `orders`, `order_items`** currently have:
- `USING (true)` on SELECT — any authenticated user can read any kitchen's data
- `WITH CHECK (true)` on INSERT — any authenticated user can insert into any kitchen

These need to be replaced with kitchen-scoped policies.

#### Recommended replacement for `orders` (apply same pattern to `customers`, `order_items`):

```sql
-- Drop open policies first
DROP POLICY IF EXISTS "Allow read" ON orders;
DROP POLICY IF EXISTS "Allow insert" ON orders;

-- Scoped SELECT: only own kitchen's orders
CREATE POLICY "kitchen_select" ON orders
  FOR SELECT USING (
    kitchen_id = (
      SELECT kitchen_id FROM profiles WHERE user_id = auth.uid()
    )
  );

-- Scoped INSERT: only into own kitchen
CREATE POLICY "kitchen_insert" ON orders
  FOR INSERT WITH CHECK (
    kitchen_id = (
      SELECT kitchen_id FROM profiles WHERE user_id = auth.uid()
    )
  );
```

**Note for anonymous customer order placement:** The `customers` and `orders` INSERT policies must also allow `anon` role (unauthenticated customers placing orders), scoped by passing `kitchen_id` explicitly.

#### Tables with correct policies:
- `kitchens` — owner-only via `owner_user_id = auth.uid()`
- `profiles` — user-own via `user_id = auth.uid()`
- `menu_items` — kitchen-scoped via profile join
- `notifications_log` — kitchen-scoped via profile join
- `ingredients`, `stock_logs` — kitchen-scoped via profile join

## DATA FLOW DIAGRAMS

### Order Placement
```
Customer fills cart (CartSheet)
  → SELECT customers WHERE phone = ? (check existing)
  → INSERT customers (if new)
  → INSERT orders (status='pending')
  → INSERT order_items[] (batch)
  → Redirect to /[slug]/track?order_id=UUID
  → localStorage.setItem('last_order_{slug}', { orderId, status })
```

### Order Status Change (Dashboard / KDS)
```
OrderBoard / OrderGrid
  → UPDATE orders SET status=?, updated_at=now() WHERE order_id IN (?)
  → Supabase Realtime subscription triggers refresh on all listeners
```

### Automation Approval
```
N8N detects trigger (low stock, win-back, upsell)
  → INSERT notifications_log (status='pending', approval_token=uuid)
  → Owner sees badge in Sidebar + card in Approvals page
  → Owner clicks "Approve"
  → POST /api/approve { token: approval_token }
  → API calls GET N8N_WEBHOOK_URL/webhook/stock-approve?token=...
  → ApprovalsBoard UPDATE notifications_log SET status='approved', resolved_at=now()
```

### Feedback Submission
```
Customer submits /[slug]/feedback form
  → POST /api/feedback (Next.js proxy)
  → API validates with Zod
  → POST N8N_WEBHOOK_URL/feedback with { order_id, kitchen_id, rating, comment }
  → N8N writes row to feedback table directly via Supabase service key
```

### Wastage Logging
```
Staff submits WastageForm (/kitchen/wastage)
  → INSERT stock_logs (change_amount = -amount, logged_by = user.id)
  → UPDATE ingredients SET current_stock = current_stock - amount
  (Two separate writes — not atomic. No transaction.)
```

## KNOWN BUGS / INCONSISTENCIES

| Issue | Location | Impact |
|---|---|---|
| Reports page queries `feedbacks` (plural) | `app/dashboard/reports/page.tsx:79` | Customer satisfaction card always shows "No feedback received yet" even when feedback exists. Fix: change `.from('feedbacks')` to `.from('feedback')` |
| `total_orders` / `total_spend` on `customers` not updated by app | `components/customer/CartSheet.tsx` | Always shows 0/0 unless N8N updates them via trigger. The Customers dashboard shows stale values. |
